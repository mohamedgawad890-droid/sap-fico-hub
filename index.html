1. What is ERP?

ERP stands for Enterprise Resource Planning — a software system that integrates all core business processes (finance, procurement, sales, HR, production) into one unified platform.

  • All departments work on the same system and share the same data in real time
  • Eliminates data silos and manual re-entry between departments
  • Provides a single source of truth for management reporting and decision making
  • Examples of ERP systems: SAP, Oracle, Microsoft Dynamics, Odoo

2. What is SAP?

SAP is the world's leading ERP software, founded in 1972 in Walldorf, Germany by five former IBM engineers: Dietmar Hopp, Hans-Werner Hector, Hasso Plattner, Klaus Tschira, and Claus Wellenreuther.

  • SAP = Systems, Applications & Products in Data Processing
  • Used by over 440,000 customers in more than 180 countries
  • Runs in 77% of Fortune 500 companies
  • Headquartered in Walldorf, Germany — listed on the Frankfurt Stock Exchange

3. SAP Version History

R/1 1972 — First version, financial accounting only
R/2 1979 — Mainframe-based, multi-currency support
R/3 1992 — Client-server architecture, global success
ECC 6.0 2005 — Enhanced R/3, still most widely used today
S/4HANA 2015 — Latest generation, in-memory HANA database
⚠️ SAP will end support for ECC in 2027. All companies are expected to migrate to S/4HANA.

6. SAP S/4HANA

S/4HANA is SAP's latest-generation ERP platform, launched in 2015. The "S/4" stands for Suite for HANA.

10. Consultant vs Key User vs End User

SAP Consultant

The technical and functional expert. Configures the system (SPRO), designs business processes, maps company requirements to SAP, trains key users, and supports go-live. Works for a partner company or as a freelancer. Needs deep SAP knowledge.

Key User

An experienced employee from the customer's side who knows both the business process and SAP well. Acts as the bridge between consultants and end users. Trains colleagues, handles basic issues, and communicates change requests to the IT/consultant team.

End User

A regular employee who uses SAP daily to do their job — posting invoices, creating purchase orders, entering timesheets, etc. Uses only the transactions relevant to their role. Does not configure or change anything in the system.

2. SAP System Landscape — Three Servers

Every SAP implementation has at least three separate environments (servers) to ensure safe development and testing before anything reaches the live system.

DEV — Development Server
Where consultants configure and build. All changes start here.
QAS — Quality Assurance / Testing Server
Testing environment. Key users and consultants test all configurations here before go-live.
PRD — Production Server
The live system. Real company data. Only tested and approved changes are moved here.
Sandbox
Free training environment — used for learning and experimentation. No rules, no consequences.
💡 Changes always flow in one direction: DEV → QAS → PRD. You never make direct changes in PRD.

4. SAP Activate Methodology — 6 Phases

SAP Activate is the current official implementation methodology used for SAP projects (replaced the older ASAP methodology). It follows 6 sequential phases.

1
Discover
Sales team presents SAP to the potential customer. Demos, proposals, and scope definition. Project is sold and approved.
2
Prepare
Project kick-off. Teams are formed, infrastructure is set up, project plan is created, and access to systems is granted.
3
Explore → Blueprint
Workshops with the client to gather all requirements. Consultants document everything in the Blueprint. Phase ends with client sign-off on the blueprint.
4
Realize
Starts only after blueprint sign-off. Consultants configure the system, build WRICEF objects, run testing cycles, and train the client's team.
5
Deploy
Final preparation for go-live. Opening balances are loaded, final master data is migrated to PRD, and cutover plan is executed.
6
Run
Go-live. The system is now live. Consultants provide Hyper Care support (first few weeks). After stabilization, a support team takes over ongoing maintenance.
📌 SAP Activate replaced the old ASAP Methodology. If you see ASAP referenced, it is the legacy approach — Activate is the current standard.

9. WRICEF — Custom Development Objects

During the Realize phase, consultants identify and build all custom objects needed beyond standard SAP. These are classified under the WRICEF framework.

WWorkflow — Automated approval processes and notifications
RReport — Custom reports built in ABAP beyond standard SAP reports
IInterface — Data exchange between SAP and external systems

10. Realize Phase — In Detail

Step 1 — Configuration: Consultants configure SAP in DEV using SPRO (IMG). All system settings, org structure, master data rules are set up.
Step 2 — Unit Test (Internal): Each consultant tests their own module independently. This is internal — done by the consulting team without the client.
Step 3 — Integration Test (External): All modules are tested together end-to-end. Done with the client's key users present. Tests cross-module flows (e.g. P2P: MM → FI).
Step 4 — Cycle Test: Full business cycle tested across all modules simultaneously to simulate real operations.
Step 5 — Training: Consultants train the client's key users on how to use the system.
Step 6 — UAT (User Acceptance Test): The client's team tests the full system themselves. All tested cycles are documented. Client provides final sign-off before go-live.

  • System is ready for go-live

12. FI Sub-Modules

  • GL — General Ledger: Core of FI. All postings flow here. Produces Trial Balance and financial statements.
  • AP — Accounts Payable: Manages vendor invoices, payment runs, and vendor balances.
  • AR — Accounts Receivable: Manages customer invoices, collections, and customer balances.
  • Bank: House bank management, cash journal, and bank reconciliation.
  • Asset Accounting (AA): Fixed assets, depreciation runs, asset acquisitions and disposals.

13. CO Sub-Modules

  • Cost Center Accounting (CCA): Tracks costs by department or function. Answers: "How much did each department spend?"
  • Profit Center Accounting (PCA): Tracks revenues and costs by business unit. Answers: "Which segment is profitable?"
  • Internal Orders: Tracks costs for specific short-term activities or projects (events, campaigns, small capex).
  • Product Costing (CO-PC): Calculates the cost of manufactured products. Used with PP module.
  • Profitability Analysis (CO-PA): Analyzes profitability by customer, product, region, or sales channel.

14. Document Type

SAP stores each type of financial transaction in a separate document. Every journal entry is assigned a Document Type that identifies what kind of transaction it is.

  • Each Document Type has a unique 2-character key (e.g. KR = Vendor Invoice, DR = Customer Invoice, SA = G/L Account Document)
  • Controls the number range for document numbering
  • Every journal entry is assigned a unique document number linked to its document type
  • Examples: Customer invoices are stored separately from vendor invoices, which are stored separately from G/L entries
💡 Document Types make it easy to filter, search, and report on specific transaction types. You can pull all vendor invoices or all customer invoices separately.

8. Reconciliation Accounts — Key Concept

A Reconciliation Account is a GL account that is linked to a sub-ledger (AR, AP, Asset Accounting). This is one of the most important concepts in FI.

⚠️ Rule: No direct posting to Recon. Accounts

If a GL account is flagged as a Reconciliation Account → you cannot post a manual debit/credit journal entry directly to it. You must post through the sub-ledger transaction.

Example:
  • GL account is a Vendor Reconciliation Account
  • You cannot post a direct journal entry to it (Dr / Cr)
  • You must use FB60 (Enter Vendor Invoice) and select a specific Vendor
  • SAP then automatically posts to the Recon. account behind the scenes

3. ⚠️ Common Error — "Post Auto Only" + Reconciliation Account

During practice in FS00, a common error appears when two conflicting settings are activated simultaneously on the same GL account.

❌ The Conflict:
  • Post Automatically Only — prevents manual journal entries; only the system can post to this account
  • Reconciliation Account for Account Type — also prevents direct manual posting (by definition)
  • SAP issues an error because both settings block direct posting — they are redundant and contradictory when used together
✅ How to Fix It:
Case 1: You want a Reconciliation Account

Remove "Post Auto Only" — the Recon. Account already blocks direct posting automatically. No need to add Post Auto Only.

Case 2: You want system-only posting

Use "Post Auto Only" only — do NOT set Reconciliation Account type. Used for clearing accounts (e.g. GR/IR account).

4. GL Account Management Functions

From within FS00, the following management actions are available on any GL account:

Block
Temporarily freezes the account. No new postings can be made to a blocked account. Used when an account needs to be reviewed or is no longer in use. Reversible at any time.
Select for Deletion
Marks the account for deletion. The account is not deleted immediately — it is flagged and physically removed later by a separate archiving run. The account must have zero balance and no open items.
Change Company Code
Transfer or update the Company Code-level settings for this GL account. Useful when extending an account to a new Company Code without recreating it from scratch.
T-Code: FS15 — Copy GL Account Master Data

Use FS15 to copy an existing GL account's master data to a new account number. This saves time and ensures consistency — instead of entering all fields manually, you copy from a similar account and adjust only what needs to change.

5. GL Document Posting — T-Code FB50

FB50 is the standard transaction for posting manual journal entries directly to GL accounts. It consists of two parts: Header Data and Line Items.

Part A — Document Header

e.g. invoice date, receipt date

The date posted to the accounting period
Determines which fiscal period is affected

e.g. invoice number, contract number

Part B — GL Account Field (Line Items)

In the line items area, you search for the GL Account by description or account number. SAP allows multiple search methods (F4 help): by account number, short text, or long text.

Part C — Action Buttons (Top of Screen)
Simulate
Preview the journal entry (Debit = Credit check) before posting. No document is created. Use this to verify the entry is correct.
Hold
Save the entry as a personal draft without posting. Only the same user can retrieve and continue it. Does not appear in any reports or balances.
Park
Save the document in a shared draft state. Other users (e.g. supervisor) can open, review, and complete it. Appears in parked document reports but does not update GL balances yet.
Post
Final, irreversible posting. Creates a permanent Document Number, updates all GL balances, and the entry becomes part of the audit trail. Cannot be deleted — only reversed.
Save as Completed
Equivalent to Post — saves and posts the document as final. Used when the entry is complete and ready for immediate posting.
⚠️ Hold vs Park — Key Difference: Hold = private draft (only you can open it). Park = shared draft (any authorized user can open, edit, and post it). Both do NOT update GL balances.

3. FB08 — Reverse Document

FB08 is used to reverse (cancel) a posted document. In SAP, you cannot delete a document once it has been posted — the audit trail must remain intact.

⚠️ SAP Rule: You can never delete a posted accounting document. Instead, you record a reversal of the wrong document, which creates a mirror-image entry that cancels it out — then you post a new correct document.
  • Enter the Document Number, Company Code, and Fiscal Year of the document to be reversed
  • Enter a Reversal Reason (e.g. 01 = Reversal in current period) — this field is mandatory
  • Enter the Posting Date for the reversal entry
  • Click Post — SAP creates a new document that is the exact mirror of the original, effectively cancelling it
Additional Options in FB08
  • Display Before Reversal — view the document first before executing the reversal
  • Document List — view a list of documents to select for reversal
  • Mass Reversal — reverse multiple documents at once
Why This Matters
  • SAP maintains a complete, unbroken audit trail for compliance and reporting
  • Every reversal creates a traceable paper trail — both the original and the reversal remain in the system
  • Auditors can see the full history of corrections

4. FBV0 — List of Parked Documents

💡 Park vs Hold: Park (FBV0) = shared draft, visible to other authorized users. Hold = private draft, only accessible by the user who created it.

General Ledger The Consultant Layer

You can post, display, reverse, and report. Now learn why it works that way and what happens when it breaks.

1. The Real GL Workflow What Accountants Do Every Month

In your course you learned individual T-codes. In a real company, GL is driven by a monthly closing process a sequence of steps that must happen in the right order. Missing one step creates errors that are painful to fix.

The Monthly GL Closing Sequence
1
Complete All Postings for the Period
All invoices, payments, journal entries must be posted before closing begins. Nothing can be added after the period closes.
2
Post Recurring Entries F.14
Automatically posts pre-defined recurring documents (rent, insurance, depreciation accruals). Must run before closing.
3
Post Accruals and Deferrals FBS1 / F.81
Accruals are expenses incurred but not yet invoiced. FBS1 posts them with a reversal date; F.81 auto-reverses them next period.
4
Run Foreign Currency Revaluation FAGL_FC_VAL
Revalues open items in foreign currencies at the current exchange rate. Required by IFRS/GAAP if the company has forex transactions.
5
Reconcile Sub-Ledgers to GL
Verify AP totals (FBL1N totals = GL account 210100 balance), AR totals, Asset values. Any difference means a posting error that must be found.
6
Close the Posting Period OB52
Lock the period so no more postings can go into it. Only done after all reconciliation is complete. Special periods 13-16 remain available for authorized year-end adjustments.
Consultant reality: In most companies this entire sequence must be completed within 3-5 business days of month-end. Knowing the sequence protects you from making expensive errors under time pressure.

2. Posting Period Control The Gate of GL

One of the most common calls an FI consultant gets: "I cannot post a document, it says the posting period is not open." Understanding this means you solve it in 60 seconds.

  • Every Company Code is assigned a Posting Period Variant (e.g. K4, Z001). This variant defines which periods are open.
  • T-code OB52 is where you open and close periods. You set it per Account Type: + (all types), A (Assets), D (Customers), K (Vendors), M (Materials), S (GL accounts).
  • Each row defines a From Period and To Period that are open. Anything outside that range is blocked.
  • Periods 13, 14, 15, 16 are special adjustment periods used only by senior accountants for year-end adjustments after regular period 12 is closed.
What a student knows

When I get a period error I need to ask someone to fix it.

What a consultant knows

Go to OB52, find the correct Posting Period Variant, check if the period is in the From-To range for that account type, open it if authorized.

Real Scenario: Wrong Period Posting

A junior accountant posted an invoice dated January in February because the period was still open. The January P&L is now wrong.

Consultant approach: Step 1 - Identify the document in FB03. Step 2 - If January is still open in OB52, reverse with FB08 and repost with correct January date. Step 3 - If January is already closed, reverse in February then post a period allocation correction entry. Step 4 - Document the correction and notify the CFO.

3. Recurring Entries and Accruals The GL Work Nobody Teaches

In real companies a large portion of GL postings happen automatically through recurring entries and accruals. A consultant who does not know these will be lost on any real project.

Recurring Entry Documents
  • FBD1 Create a Recurring Entry document. Define the GL accounts, amounts, first run date, last run date, and interval (monthly, quarterly). SAP stores this as a template and does NOT post immediately.
  • F.14 Execute Recurring Entries every month-end. SAP finds all templates due and posts them automatically as real accounting documents.
  • Common uses: monthly rent, insurance premiums, depreciation accruals, management fee allocations.
Consultant tip: Always check if F.14 has been run before closing a period. Many month-end errors come from recurring entries not executed.
Accrual Documents
  • FBS1 Post an Accrual document. You post the entry AND set a Reversal Date in the same screen. The document posts normally but is flagged for automatic reversal.
  • F.81 Execute Scheduled Reversals at the start of the next period. SAP automatically reverses all FBS1 documents whose reversal date has passed.
  • Example: Electricity for December not yet billed. Post Dr Electricity Expense Cr Accrued Liabilities on Dec 31 with reversal date Jan 1. F.81 reverses automatically. When real invoice arrives it posts normally.
Real Scenario: Accrual Not Reversed

An accountant posted an accrual in December using FB50 (a regular journal entry, not FBS1). Nobody reversed it in January. The real invoice came in February and was posted. Result: the expense was counted twice. Prevention: Always use FBS1 for accruals. FBS1 forces you to set a reversal date and F.81 handles it automatically.

5. The 6 Configuration Objects Behind GL

Every time you posted a document in FB50, six configuration objects worked silently in the background. When something goes wrong, one of these six is always the cause.

OB13
Chart of Accounts
Defines the GL account structure

Every Company Code must be assigned to a Chart of Accounts. If wrong: GL accounts do not exist and no postings are possible.

OBBO / OB52
Posting Periods
Controls which periods are open for posting

OBBO creates the variant. OBB8 defines open periods. OB37 assigns the variant to a Company Code. If wrong: users cannot post or can post to closed periods (audit risk).

OBA7
Document Types
Defines document type properties

Configures which account types each document type allows, reversal behavior, and required fields. If wrong: users cannot post invoices or documents cannot be reversed.

FBN1
Number Ranges
Assigns number sequences to document types

Each document type gets a number range per year. SAP increments automatically. If wrong: number range exhausted and posting fails, or two types share a range causing document number overlaps.

OBC4
Field Status Groups
Controls which fields appear when posting

Defines for every field whether it is Suppressed, Optional, or Required. The posting key also has a field status and the more restrictive one wins. If wrong: field status conflict error blocks all postings to the account.

OBA4
Tolerance Groups
Controls how much a user can post

Defines maximum document amount, maximum line item amount, and maximum cash discount. If wrong: junior accountant can post a 10 million EGP document by mistake, or senior accountant cannot post a large legitimate invoice.

6. The 5 GL Errors Every Consultant Faces on Day One

These are not theoretical. Every FI consultant encounters these in the first 90 days on a project.

Error 1: Posting period not open

Cause: Period is closed in OB52.
Solution: Go to OB52, find the Posting Period Variant, open the required period, user retries.
Prevention: Agree a period opening/closing calendar with the client at project start.

Error 2: Field status conflict

Cause: Field Status Group on the GL account and the Field Status of the Posting Key conflict. One says Required, the other says Suppressed. SAP cannot resolve it.
Solution: Go to OBC4, find the Field Status Group of the account, align the conflicting field to Optional in both.
Example: Cost Center is Required on the posting key but Suppressed on the account.

Error 3: Not authorized for this amount

Cause: Document amount exceeds the user Tolerance Group limit in OBA4.
Solution: Increase the tolerance for this user group (requires authorization) or have a senior user post it.
Consultant note: Never increase tolerances without written approval from the financial controller. This is an audit control.

Error 4: Cannot post to this account directly

Cause: The user is trying to post directly to a Reconciliation Account (210100 Vendors, 110100 Customers, or an Asset account). Direct posting to reconciliation accounts is blocked by design.
Solution: Use the correct T-code. FB60 for vendor invoices, FB70 for customer invoices. Never try to bypass via F-02 with posting key 40/50 directly to a recon account.

Error 5: Document cannot be reversed

Cause: The document has already been cleared (e.g. an invoice that was paid). Cleared documents cannot be reversed directly.
Solution: Use FBRA (Reset Clearing) to unlink the payment from the invoice, then reverse the original document with FB08, then re-clear or re-pay as needed.
Warning: This affects the vendor/customer account balance. Always verify the impact before doing it in production.

8. Year-End GL Closing The Most Critical Process

Two specific T-codes that every GL consultant must know for year-end:

FAGLGVTR Balance Carryforward

Carries the closing balances of Balance Sheet accounts (Assets, Liabilities, Equity) forward to period 0 of the new fiscal year. P&L accounts are zeroed out and their balance is transferred to the Retained Earnings account. This is why in January of a new year, Balance Sheet accounts have an opening balance and P&L accounts start at zero.

FAGLGVTR can be run in simulation mode during the year. The final run must happen before any postings in the new year are reported. Running it late means the new year starts with wrong opening balances.
OB52 Special Periods 13 to 16

After period 12 (December) is closed, auditors and senior accountants may still need to post year-end adjustments. SAP provides special periods 13, 14, 15, and 16 opened separately in OB52 and accessible only to authorized users. This allows year-end audit adjustments without reopening the regular December period.
